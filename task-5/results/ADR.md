# ADR-005. Мониторинг, логирование и оповещение Spring Batch ETL

- **Статус:** Accepted
- **Дата:** 2026-04-25
- **Авторы:** Вячеслав Стеблецов
- **Контекст:** Задание 5 проектной работы 5 спринта MSA.

## Решение

Для мониторинга Spring Batch-приложения используется связка Spring Boot Actuator, Micrometer, Prometheus и Grafana.

Приложение публикует метрики через endpoint `/actuator/prometheus`. Prometheus периодически собирает метрики, а Grafana используется для визуализации состояния batch job, step execution, JVM и инфраструктурных показателей.

Так как целевая production-модель предполагает запуск ETL как Kubernetes CronJob, Spring Batch-приложение является short-lived процессом: оно стартует по расписанию, обрабатывает CSV-файл и завершает работу. Поэтому отсутствие постоянно запущенного контейнера не считается инцидентом само по себе.

Для локальной демонстрации приложение запускается в Docker Compose и отдаёт метрики через `/actuator/prometheus`. Для production-варианта с Kubernetes CronJob дополнительно рекомендуется использовать Pushgateway или метрики Kubernetes Job через kube-state-metrics, чтобы хранить результат последнего batch-запуска после завершения pod.

Для логирования используется централизованная схема **Filebeat → Logstash → Elasticsearch → Kibana**.

Spring Batch-приложение пишет структурированные логи в `stdout` контейнера. Docker сохраняет эти логи в своей файловой системе, после чего Filebeat считывает их напрямую из docker container logs:

/var/lib/docker/containers/*/*.log

Далее Filebeat отправляет события в Logstash, Logstash обрабатывает их и индексирует в Elasticsearch.

Kibana используется для поиска и анализа логов: ошибок обработки CSV, статусов job/step, причин skip/retry, а также конкретных запусков по `jobId`, `stepName`, `status` и тексту исключения.

Такой подход соответствует best practices контейнерных приложений и позволяет отказаться от файлов логирования внутри контейнера. В Kubernetes аналогичная схема реализуется через Filebeat или Fluent Bit DaemonSet, который собирает stdout/stderr container logs со всех worker nodes.

## Обоснование выбора метрик

Основные бизнес-метрики batch-процесса:

- длительность Job и Step;
- количество прочитанных строк;
- количество записанных строк;
- количество пропущенных строк;
- количество ошибок;
- статус завершения Job;
- timestamp последнего успешного запуска.

Эти метрики позволяют контролировать SLA: обработка 2 000 строк не должна превышать 30 секунд. Также они позволяют выявлять деградацию производительности, рост количества невалидных строк и падения batch job.

Дополнительно собираются технические метрики JVM, CPU и памяти:

- `jvm_memory_used_bytes`;
- `jvm_memory_max_bytes`;
- `process_cpu_usage`;
- `system_cpu_usage`.

Они нужны для диагностики проблем с производительностью и подготовки к запуску в Kubernetes/GKE.

Для batch/CronJob-сценария наиболее важны не метрики постоянной доступности приложения, а метрики результата последнего запуска:

- `stock_etl_last_run_status`;
- `stock_etl_last_success_timestamp_seconds`;
- `stock_etl_last_run_duration_seconds`;
- `stock_etl_last_run_skip_count`.

В локальном стенде эти метрики могут быть реализованы как custom Micrometer metrics. В production они должны сохраняться после завершения Job через Pushgateway или собираться на уровне Kubernetes Job.

## Обоснование логирования

Для логирования выбран Filebeat, так как он является легковеским агентом, не требует изменения бизнес-кода приложения и хорошо подходит для контейнерной инфраструктуры.

Вместо записи логов в файл внутри контейнера используется вывод в `stdout`. Это более корректно для Docker/Kubernetes, потому что:

- контейнер остаётся stateless;
- не требуется volume для application logs;
- сбор логов не зависит от файловой структуры приложения;
- решение проще масштабировать;
- подход совместим с Kubernetes DaemonSet-моделью.

Elasticsearch выбран как централизованное хранилище логов, Kibana — как интерфейс анализа. Это позволяет DevOps/SRE быстро находить ошибки конкретного batch job по `jobId`, `stepName`, `status`, `skipCount`, `readCount`, `writeCount` и тексту исключения.

## Оповещения

Оповещения настраиваются в Grafana на основе Prometheus datasource. Prometheus rules используются как дополнительный способ описания alert conditions в конфигурации.

Так как Spring Batch ETL запускается по расписанию и завершает работу после обработки файла, алерт `up == 0` не используется как основной бизнес-алерт. Для CronJob отсутствие running container после завершения обработки является штатным состоянием.

Основные алерты строятся вокруг результата последнего запуска:

- `StockEtlLastRunFailed` — последний запуск завершился ошибкой;
- `StockEtlSlowRun` — длительность последнего запуска превысила SLA 30 секунд;
- `StockEtlHighSkipCount` — в последнем запуске были skipped records.

Grafana выбрана для алертов, потому что она объединяет визуализацию, настройку alert rules и каналы уведомлений в одном интерфейсе. Prometheus rules остаются резервным вариантом для инфраструктурных и batch-алертов.
