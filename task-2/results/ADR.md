# ADR-2. Выбор решения для ежедневной генерации B2B-прайс-листов

## Общая информация

1. **Родительский артефакт** — Архитектура онлайн-магазина (микросервисы в облаке, K8s).
2. **Автор ADR** — Вячеслав Стеблецов.
3. **Статус** — Принят.
4. **Согласующие** — Архитектор решения, Tech Lead backend, DevOps / SRE, Product Owner B2B-направления.

## Контекст

Онлайн-магазину ежедневно **ровно в 06:00** требуется генерировать персональные CSV/XLS-прайс-листы для B2B-клиентов на основе текущего состояния БД и выкладывать их в объектное хранилище для последующей выдачи клиентам (прямая ссылка, email, партнёрский портал).

**Исходные данные (PostgreSQL):**

| Таблица | Объём строк |
|---|---|
| `products` | 5 000 – 10 000 |
| `categories` | 50 – 200 |
| `clients` | 100 – 500 |
| `client_prices` | 10 000 – 20 000 |

**Характеристики задачи:**

- триггер — фиксированное CRON-расписание (06:00 ежедневно);
- объём выгрузки на один запуск — **5 000–10 000 строк** на клиента;
- на каждого B2B-клиента формируется отдельный файл (итого 100–500 файлов в сутки);
- **нет сложной логики обработки**: JOIN четырёх таблиц + форматирование в CSV/XLS;
- целевая инфраструктура — Kubernetes в облаке, уже эксплуатируются сервисы магазина;
- доступны централизованные логирование (ELK) и мониторинг (Prometheus + Grafana).

**Ключевые требования:**

- CRON-расписание «из коробки»;
- минимальная эксплуатационная сложность и стоимость владения;
- естественная интеграция с существующими микросервисами в K8s;
- наблюдаемость (логи, метрики, алерты при сбое);
- возможность повторного запуска / retry при сбое;
- горизонтальная масштабируемость (рост числа B2B-клиентов).

**Вне scope ADR:** схема данных прайс-листа, формат рассылки клиентам, политика хранения файлов в объектном хранилище, права доступа B2B-клиентов.

## Рассмотренные варианты

### Вариант №1 «Spring Batch»

Java/Spring-фреймворк для пакетной обработки. Модель `Job → Step → Reader/Processor/Writer`, рестартуемость шагов, chunk-oriented processing, встроенный `JobRepository` в БД. CRON — через Spring `@Scheduled` или внешний шедулер. Нативен для команд со Spring-стеком.

### Вариант №2 «Apache Airflow»

Оркестратор DAG-пайплайнов на Python. Богатый функционал: retry, ветвления, sensors, email-уведомления, Web UI, провайдеры для Postgres/S3. Разворачивается как managed-сервис (Composer/MWAA) или через Helm-chart (Scheduler + Webserver + Workers + metadata DB).

### Вариант №3 «Kubernetes CronJob»

Нативный примитив Kubernetes: CRON-расписание запускает `Job`, который создаёт `Pod` с контейнером приложения. Приложение — обычный контейнер (Python/Go/Java) с одной-двумя командами: прочитать из Postgres, записать CSV в S3/GCS. Retry через `backoffLimit`, логи — в stdout (ELK), метрики — через Pushgateway или sidecar.

### Вариант №4 «Apache Spark»

Распределённый движок обработки данных. Поддерживает JDBC-источники, запись CSV/Parquet, масштабирование по кластеру. Запускается через Spark-on-K8s или managed-сервисы (Dataproc/EMR). Расписание — через внешний шедулер (Airflow, K8s CronJob).

## Сравнение альтернатив

### Сводная таблица по критериям задания

Оценка: **✅ — сильная сторона**, **🟡 — приемлемо / с оговорками**, **❌ — слабая сторона**.

| Критерий | Spring Batch | Apache Airflow | K8s Job (CronJob) | Spark |
|---|---|---|---|---|
| Наличие конфигурации CRON-расписания | 🟡 Через `@Scheduled` внутри приложения или внешний шедулер; сам Spring Batch расписанием не управляет. | ✅ CRON-выражение в DAG — встроенная функция. | ✅ Поле `schedule` в `CronJob` manifest — нативно. | ❌ Собственного шедулера нет, требуется внешний (Airflow/CronJob). |
| Сложность реализации логики по обработке данных | 🟡 Средняя: нужно описать `Job/Step/Reader/Writer`, даже если логика тривиальна — получается «over-engineering» для простой выгрузки. | 🟡 Нужно описать DAG и оператор (`PostgresOperator` + `S3Hook`) — избыточно для одной задачи. | ✅ Минимальная: один Python/Go-скрипт на ~50–100 строк (SQL + запись CSV в S3). | ❌ Высокая: для 10k строк Spark-оверкилл, накладные расходы на инициализацию драйвера/исполнителей превышают полезную работу. |
| Ресурсоёмкость решения | 🟡 JVM-процесс постоянно «живой» при `@Scheduled`; память на JVM и Spring-контекст. | ❌ Постоянно работают Scheduler + Webserver + metadata DB (Postgres) — заметные ресурсы даже без запусков. | ✅ Под создаётся только на время выполнения (~секунды–минуты), остальное время — 0 ресурсов. | ❌ Driver + executors, JVM, кластерные ресурсы — неоправданно для 10k строк. |
| Масштабируемость под нагрузкой и сложность реализации | 🟡 Партиционирование шагов (`PartitionHandler`) — есть, но требует ручной настройки. | ✅ Dynamic task mapping, параллельные таски, Celery/Kubernetes executor. | ✅ Параллелизм через `parallelism`/`completions` в `Job`; на каждого клиента — свой Pod при необходимости. Рост объёма покрывается горизонтально. | ✅ Нативная распределённость, но бесполезна при текущем объёме. |
| Сложность развёртывания в облаке и интеграция с микросервисной архитектурой | 🟡 Приложение деплоится как обычный сервис, но нужно выделять отдельный инстанс под batch, чтобы не мешать онлайн-трафику. | ❌ Либо managed-сервис (Composer/MWAA) — отдельная оплата и vendor-lock, либо Helm-chart с 4+ компонентами и Postgres. | ✅ Нативно для K8s: один манифест `CronJob`, общий CI/CD с другими микросервисами, те же practices по Secrets/ConfigMap/RBAC. | ❌ Требуется Spark-оператор в кластере или managed-Spark, отдельная экспертиза. |
| Удобство интеграции с системами логирования и мониторинга | 🟡 Через Micrometer + stdout; требуется аккуратная настройка. | ✅ Web UI, StatsD/Prometheus exporter, но ещё одна «панель» в дополнение к Grafana магазина. | ✅ stdout → ELK (как у остальных сервисов), Prometheus метрики через `kube-state-metrics` (`kube_job_status_failed`), алерты в Grafana/Alertmanager. | 🟡 Spark UI, History Server — отдельная инфраструктура. |

### Текстовые выводы

- **Airflow и Spark** — мощные инструменты для сложных пайплайнов и больших объёмов, но в данном кейсе (простой JOIN на 10k строк, одна задача раз в сутки) они создают непропорциональные эксплуатационные и ресурсные накладные расходы.
- **Spring Batch** уместен, если бы в компании уже был Spring-стек и требовались рестартуемость/chunk-processing для сотен миллионов записей. Здесь такого контекста нет.
- **Kubernetes CronJob** попадает в задачу «точно по размеру»: расписание, запуск по требованию, нулевое потребление в простое, единая модель деплоя с остальными микросервисами.

## Принятое решение

Принят **Вариант №3 «Kubernetes CronJob»**.

### Обоснование

1. **Соответствие масштабу задачи.** Объём 5–10 тыс. строк и одна выгрузка в сутки не требуют распределённого движка (Spark) или оркестратора DAG (Airflow). Добавлять их — значит оплачивать инфраструктуру и сложность, которыми мы не пользуемся.
2. **Нативность для целевой среды.** Инфраструктура магазина — K8s. `CronJob` — штатный примитив платформы: тот же CI/CD, Helm-чарты, Secrets, ServiceAccount/RBAC, NetworkPolicy, что и у остальных микросервисов. Не нужна отдельная экспертиза.
3. **Ресурсная эффективность.** Pod живёт только во время выполнения (секунды–минуты). В остальное время потребление ресурсов — 0, в отличие от Airflow (постоянно работающие Scheduler/Webserver/metadata DB) и Spring Batch (постоянный JVM-процесс).
4. **Наблюдаемость «из коробки».** Логи из stdout уходят в ELK, статусы `Job`/`CronJob` собираются `kube-state-metrics` → Prometheus → Grafana/Alertmanager. Отдельной UI-панели не требуется.
5. **Надёжность.** `backoffLimit`, `activeDeadlineSeconds`, `startingDeadlineSeconds`, `concurrencyPolicy: Forbid` покрывают типовые сценарии retry и защиты от параллельных запусков. При необходимости идемпотентность обеспечивается на уровне ключа объекта в S3 (дата + client_id).
6. **Масштабируемость при росте.** При увеличении числа B2B-клиентов параллелизм достигается через `parallelism`/`completions` в `Job` (один Pod на клиента или батч клиентов) без смены архитектуры. Если в будущем задача усложнится (множество источников, event-triggers, сложные DAG), решение безболезненно «поднимается» до Airflow — `CronJob` легко заменить на `DAG` с тем же контейнером.
7. **Стоимость владения.** Минимальная: нет дополнительных компонентов (metadata DB, брокер, UI), нет отдельной команды поддержки.
